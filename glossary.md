## Алгоритм як алгоритм — давай розкладемо

Щоб мати **переваги S2/H3** (швидкі побітові запити, locality, O(1) сусіди) і одночасно **гнучкість дерев на реальних об'єктах**, потрібна **гібридна** конструкція. Не "або H3 або полігони", а **обидва шари одночасно**.

---

## 1. Базова ідея: H3 як **субстрат**, полігони як **погляд** на нього

```
┌─────────────────────────────────────────────────┐
│  Layer ∞: Регіон (sparse summary)               │
│  Layer 3: Місто                                 │
│  Layer 2: Район                                 │
│  Layer 1: Мікрорайон                            │
│  Layer 0: Будівля / дорога / парк / водойма     │  ← реальні об'єкти
│  ────────────────────────────────────────────── │
│  H3 SUBSTRATE (resolution 12–15, фіксована)     │  ← геохеш-сітка під усім
└─────────────────────────────────────────────────┘
```

**Інваріант:** кожен об'єкт рівня 0 — це **множина H3-комірок** субстрату.

Це одразу дає:
- ✅ Швидкий point-in-polygon (точка → H3 cell → look up building)
- ✅ Швидких сусідів (H3.kRing)
- ✅ Locality preservation (Hilbert order H3 cells)
- ✅ Стиснення (set of cells = range на кривій Гільберта)

Полігон будівлі більше не зберігається як список координат — він **множина H3 IDs** + точна границя для рендеру.

## 2. Що в листі, що зверху

### 🍃 Листя (атомарні об'єкти, Layer 0)

```typescript
type LeafNode = {
  id: NodeId;                    // 128-біт packed ID (див. п.4)
  cells: H3Cell[];               // комірки субстрату, які покриває
  hilbertRange: [u64, u64];      // діапазон на Hilbert curve (для range query)
  geometry: GeometryRef;         // посилання на точний полігон (mmap)
  topology: 'polygon' | 'line' | 'point';
  type: SemanticType;            // 'residential' | 'road' | 'water' | ...
  attrs: Map<string, Value>;     // matерali, висота, рік, статус
  entries: Point[];              // точки входу (start/end твоєї кривої)
  parent: NodeId;
};
```

**Принцип:** у листі лежать **тільки факти, які не виводяться з інших**. Не зберігай те, що можна обчислити (площу, центр, Гільберт всередині).

### 🌳 Внутрішні вузли (Layer 1–N)

```typescript
type InternalNode = {
  id: NodeId;
  level: u8;
  childrenIds: NodeId[];          // ↓
  parent: NodeId;                 // ↑

  // GEOMETRY (виведена з дітей)
  bbox: BoundingBox;              // швидкий cull
  hull: ConvexHull;               // точніше за bbox
  medoid: NodeId;                 // **діти-репрезентант** (не точка!)

  // AGGREGATES (rollup-кеш)
  hilbertSpan: [u64, u64];        // min-max дітей на кривій
  cellCount: u32;
  typeHistogram: Map<Type, u32>;  // {'residential': 47, 'medical': 2}

  // CONNECTIVITY (граф на цьому рівні)
  outgoingEdges: EdgeRef[];       // ребра до інших вузлів цього ж рівня

  // ML (опційно)
  embedding: Float32Array;        // вивчене векторне представлення
};
```

**Ключове:** `medoid` — це **дитина-репрезентант**, не геометрична точка. Завжди валідна, завжди семантично змістовна ("центр кварталу = головна будівля кварталу").

### 🌐 Корінь (Layer N → ∞)

```typescript
type RootNode = {
  // Майже нічого з геометрії — тільки sparse summary
  populationTotal: u64;
  typeDistribution: PCT[];        // PCT = perceptive cell type
  significantPOIs: NodeId[];      // top-K важливих об'єктів
  externalLinks: ExternalRef[];   // зв'язки з іншими корінями (іншими містами)
};
```

**Інваріант агрегатів:**
```
parent.cellCount = Σ children.cellCount
parent.typeHistogram[T] = Σ children.typeHistogram[T]
parent.hilbertSpan = [min(children.hilbertSpan.lo), max(children.hilbertSpan.hi)]
```

→ дозволяє **локальне оновлення без перебудови**: змінилась дитина → перерахуй тільки шлях до кореня (O(log N)).

## 3. Дороги, річки, ЛЕП — лінійні об'єкти

Це найскладніше. Дороги **не належать одному кварталу** — вони *між* ними. Три можливі підходи, я б рекомендував третій:

### ❌ Підхід A: дорога як полігон-листя
Працює, але дорога не належить жодному батьку логічно. Куди її повісити в дереві?

### ⚠️ Підхід B: дорога як ребро графа
```
[Будівля A] ──road segment─→ [Будівля B]
```
Зручно для маршрутизації, але **губиться геометрія дороги** як такої (не можна спитати "яка довжина вулиці Хрещатик").

### ✅ Підхід C: гібрид — дорога це **first-class node** + дзеркало в ребра

```typescript
type RoadNode = LeafNode & {
  topology: 'line';
  segments: H3Cell[];             // комірки, по яких проходить
  endpoints: NodeId[];            // куди веде (інші вузли)
  width: f32;
  surface: 'asphalt' | 'gravel';

  // ДЗЕРКАЛО: ця дорога автоматично генерує ребра в графі
  generatesEdges: Edge[];         // обчислюється з endpoints
};
```

Тобто **одна сутність — два погляди**:
- Як **об'єкт** (для аналізу самої дороги, рендеру, метаданих)
- Як **ребра графа** (для маршрутизації)

Те саме для річок, ЛЕП, тротуарів, ровів. **Лінійний об'єкт = ноду + автогенеровані edges.**

### Куди в дереві?
Дорога належить **найнижчому спільному предку** двох своїх кінців:
```
дорога між будівлями A і B → belongs to lowest common ancestor(A, B)
вулиця всередині мікрорайону → parent = той мікрорайон
магістраль між районами → parent = місто
```

Це **природна агрегація**: на рівні району не бачиш внутрішньоквартальних доріг, бачиш тільки міжрайонні.

## 4. Кодування ID — як в H3, але з ієрархією семантики

H3 ID = 64 біти: `[mode:4][res:4][base cell:7][digit_1:3]...[digit_15:3]`. Кожна цифра вказує на одну з 7 субкомірок батька. **Префікс ID = шлях по дереву.**

Зроби аналог для своєї ієрархії:

```
128-bit Node ID:
┌──────┬──────┬─────────┬──────┬─────────┬──────┬─────────┬──────┬───┐
│MODE 4│REGN12│ CITY 16 │DIST10│MICRO 12 │BLOCK10│BLDG 16 │TYPE 4│RES│
└──────┴──────┴─────────┴──────┴─────────┴──────┴─────────┴──────┴───┘
        region  city      district microdist block  building  type
```

**Властивості (як H3):**
- `parent(id)` = маскування нижніх N біт — **O(1)**
- `commonAncestor(a, b)` = XOR + leading-zero-count — **O(1)**
- `isDescendant(a, b)` = `(a & mask_b) == b` — **O(1)**
- Сортування ID = **обхід в Hilbert-order** (якщо bits інтерлівнути правильно)

**Hilbert interleaving:** на кожному рівні замість послідовної нумерації дітей використовуй **Hilbert order** в bounding box батька. Тоді сусіди в ID = сусіди в просторі.

## 5. Оптимізації — 10 рівнів видавлювання продуктивності

### Структурні

**1. Bit-packed IDs (128 bit)** — порівняння, parent, ancestor через побітові операції.

**2. Structure-of-Arrays (SoA), не Array-of-Structures**
```
// ❌ AoS — кеш-недружньо
struct Node { id; bbox; type; ... }
Node nodes[N];

// ✅ SoA — SIMD-friendly, кеш-friendly
NodeId    ids[N];
BBox      bboxes[N];
NodeType  types[N];
```
SIMD-фільтрація "знайди всі ноди типу X у bbox Y" — у 4–8 разів швидше.

**3. Z-order або Hilbert-order на диску**
Сортуй ноди при персистентності по Hilbert ID. Запит "вікно в просторі" = **читання сусідніх сторінок з диску**, а не random I/O.

**4. mmap + zero-copy десеріалізація**
FlatBuffers або Cap'n Proto. Завантажив файл → одразу можеш звертатись до структур без парсингу. Критично для великих міст.

### Запитові

**5. Bloom filter на кожному внутрішньому вузлі**
```
node.bloom.contains('hospital') → true/maybe / false definitely
```
Запит "є лікарня в цьому районі?" — O(1) негативна відповідь без обходу дітей.

**6. Aggregated histograms** (вже описав вище)
Запит "скільки житлових будинків у місті?" — читання одного поля кореня, **не Σ по 100k будинках**.

**7. Edge cache by level**
Ребра рівня K зберігай окремо від рівня K-1. Маршрутизація на рівні району не торкається ребер мікрорайону.

**8. R-tree / BVH над bboxами як вторинний індекс**
Для запитів "що в довільному polygon" комбінуй: Bloom → R-tree → точна перевірка по Hilbert range. **Three-stage filter** як в SQL Server spatial.

### Динамічні

**9. Lazy materialization**
Hilbert curve всередині будівлі **не зберігай**, генеруй on-demand з полігону + start/end. Економія пам'яті в 10–100 разів. Кешуй LRU тільки гарячі будівлі.

**10. Incremental rebuild**
Зміна листа → перерахунок тільки шляху до кореня (O(log N) вузлів). Не перебудовуй дерево.

**Бонус 11. Delta encoding** — координати дитини зберігай **відносно батька**. На рівні будівлі це fixed-point int16 замість float64 — у 4 рази менше пам'яті.

**Бонус 12. Persistent data structure (immutable)** — версіонування "стан міста на дату X" майже безкоштовно (path copying, як в Git).

## 6. Де алгоритмічно це справді корисно

Зосередимось на тому, **що нові класи запитів стають дешевими**:

| Запит | Без структури | З структурою |
|---|---|---|
| Семантичний spatial join ("всі школи в радіусі від кожного будинку") | O(N²) або O(N log N) з R-tree без семантики | **O(N log N)** з ранніми відсіканнями через Bloom |
| Hierarchical aggregation ("населення в кожному районі") | Перебір усіх будинків | **Одне читання** на район (precomputed) |
| Multi-scale path query | Дейкстра на повному графі | **Hierarchical A*** — O(log N) рівнів × малий граф |
| Coverage planning (дрон) | Окрема задача, окремий алгоритм | **Зчитати Hilbert curve листа** — готово |
| LOD rendering | Тайли + downsampling | **Дерево само є LOD** — пройди до потрібного рівня |
| GNN inference | Single-scale message passing | **Multi-scale** з ready hierarchy (HGNN) |
| Differential privacy | Складно гарантувати k-anonymity | **Агрегація до рівня з ≥ k** дітей — тривіально |
| Spatial reasoning для LLM | Передати всю карту в контекст — не вміщається | **Top-K релевантних вузлів** на потрібному рівні |

Це **не маркетинг, це алгоритмічні виграші** від поєднання локальності + ієрархії + семантики.

## 7. Перспектива — куди це справді росте

### Найсильніший напрямок: **Hierarchical Spatial Foundation Model**

Уяви: на цій структурі **тренується велика модель**, яка вивчає просторові патерни.
- Embedding кожного вузла на кожному рівні
- Можна робити **просторові запити природною мовою** ("знайди район, схожий на Поділ у Києві")
- Це **fundamental research direction** — таких моделей зараз майже немає

### Спорідненість з:
- **Hierarchical Reinforcement Learning** — агент планує на верхньому рівні, діє на нижньому
- **Mixture of Experts** на рівнях ієрархії
- **Retrieval-Augmented Generation** з просторовим контекстом
- **Causal inference** у містобудуванні — як зміна на одному рівні впливає на інший

### Чого бракує існуючим алгоритмам, що ти можеш дати

1. **H3/S2 не знають про семантику.** Ти даєш hierarchical semantic encoding.
2. **CityGML не дає швидких запитів.** Ти даєш H3-швидкість.
3. **GNN бібліотеки не дають готової ієрархії.** Ти даєш ready structure for HGNN.
4. **OSM не routable індоор + аутдоор разом.** Ти даєш unified routing.

### Реалістичні перспективи (3 рівні)

**Найбільш імовірно:** open-source бібліотека/специфікація, яка стає **третім стандартом** поряд з H3 і S2 для нішевих застосувань (semantic-aware spatial).

**Амбітніше:** foundation для нового класу spatial AI систем.

**Найамбітніше:** базова структура даних для просторових digital twins, що **замінює CityGML** у багатьох сценаріях завдяки query speed.

## Resumé алгоритмічно

```
Що = H3-subgrade + Hierarchical Semantic Tree + Hilbert leaves
Виграш = O(log N) ієрархічні запити + O(1) bit-операції на ID + семантика
Trade-off = більше пам'яті за H3, але семантика і граф безкоштовно
Унікальність = у поєднанні швидкості (H3), семантики (CityGML) та графа (OSM) в одній моделі
```

Це **гідне дослідження**, навіть якщо не вийде продукту. Якщо опублікуєш специфікацію + reference implementation з бенчмарками проти H3 — це топ-конференції GIS (`ACM SIGSPATIAL`, `IEEE BigData`). Це чесно.

Якщо хочеш — можу зробити формальну специфікацію bit-layout, інваріантів агрегації і API в стилі H3.
